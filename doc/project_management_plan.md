## 1. 서론

### 1.1 문서 목적 및 범위

본 프로젝트 계획서는 AI 기반 상황 인식 무음 커뮤니케이션 지원 시스템(SilentTalk)에 대한 전반적인 내용을 정리하여 프로젝트를 진행하는 데 있어 각 세부 사항의 지표로 삼는 것을 목적으로 한다.

본 문서는 다음과 같은 범위의 지표로 적용할 수 있다.

- 프로젝트 개요 (프로젝트 정의, 주요 기능 정리)
- 개발 계획 및 환경 (개발 절차 모형, 내용 및 일정, 소프트웨어/하드웨어 환경)
- 팀 구성 (팀 구조/역할 및 책임)
- 품질관리 (팀 미팅, 변경 사항 및 산출물 관리)
- 산출물 (산출물 정의, 작성일 및 담당자)

### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의

본 시스템은 청각장애인 및 난청 사용자를 대상으로, 카메라 기반 상황 인식과 AI를 활용하여 의사소통을 지원하는 서비스이다. 사용자의 현재 위치 및 환경을 분석하여 상황에 적합한 문장을 자동으로 추천하고, 버튼 기반 인터페이스를 통해 빠르게 의사표현을 할 수 있도록 한다. 이를 통해 타이핑 중심의 느린 의사소통 문제를 해결하고, 자연스러운 대화 흐름을 유지하는 것을 목표로 한다.

#### 1.2.2 주요 기능 설명

| ID | 기능명 | 설명 |
|----|--------|------|
| F01 | 상황/장소 인식 | 카메라 입력을 통해 카페, 병원, 학교 등 환경 자동 인식 및 상황에 맞는 문장 추천 (Computer Vision) |
| F02 | 버튼 기반 의사 표현 | 타이핑 없이 버튼 선택으로 즉시 의사 전달 지원 |
| F03 | AI 기반 대화 흐름 추천 | 대화 맥락을 분석하여 다음 문장 자동 추천 |
| F04 | 상황 맞춤 문장 생성 | 장소 + 상황에 맞는 자연스러운 문장 생성 (NLP) |
| F05 | 제스처 기반 입력 | 손동작으로 문장 선택 및 UI 조작 (Computer Vision) |
| F06 | 사용자 맞춤 학습 | 자주 사용하는 문장을 학습하여 우선 추천 |

### 1.3 프로젝트 목표

본 프로젝트는 전 세계 15억 명 이상의 청력 손실 인구와 국내 약 40만 명 이상의 청각장애인이 겪는 의사소통 문제를 해결하기 위해 제안되었다. 기존의 타이핑 기반 의사소통 방식은 일반 대화 대비 속도가 느리며, 상황을 고려하지 못하는 한계를 가진다.

이에 따라 본 프로젝트는 다음의 목표를 달성하고자 한다.

- 카메라 기반 상황 인식을 통한 실시간 맞춤형 문장 자동 추천
- 버튼 기반 인터페이스로 버튼 선택 후 1초 이내 의사 전달 가능
- AI 대화 흐름 분석을 통한 자연스러운 의사소통 지원
- 청각장애인 및 난청 사용자에 최적화된 접근성 높은 UI 제공

### 1.4 용어 정의

| 용어 | 설명 |
|------|------|
| Computer Vision (CV) | 카메라 등 이미지/영상 입력으로부터 시각적 정보를 인식·분석하는 AI 기술 |
| NLP | 자연어 처리 (Natural Language Processing). 텍스트를 이해하고 생성하는 AI 기술 |
| AAC | 보완대체의사소통 (Augmentative and Alternative Communication). 언어 장애인을 위한 의사소통 지원 기술 |
| TTS | 텍스트 음성 변환 (Text-to-Speech). 텍스트를 음성으로 출력하는 기술 |
| 버튼 기반 인터페이스 | 타이핑 없이 미리 정의된 버튼을 선택하여 의사를 표현하는 UI 방식 |
| 상황 인식 | 카메라 입력으로 현재 장소 및 환경(카페, 병원 등)을 자동으로 파악하는 기능 |

### 1.5 참조 문서

| 문서명 | 비고 |
|--------|------|
| 프로젝트 정의서 | 프로젝트 배경, 핵심 기능, 유사 소프트웨어 분석 포함 |
| 시스템 품질 요소 | 사용자·개발자·관리자 관점 품질 요소 정의 |

## 2. 개발 계획

### 2.1 개발 절차 모형

#### 2.1.1 소프트웨어 개발 절차 모형

SilentTalk는 AI 모델(CV, NLP) 연동, 사용자 인터페이스, 상황 인식 엔진 등 여러 모듈을 병행 개발해야 하며, 초기 요구사항이 완전히 확정되지 않고 사용자 피드백에 따라 기능이 점진적으로 구체화될 것으로 예상된다. 이에 따라 빠른 반복 개발 및 지속적 피드백 반영이 가능한 Agile 방법론의 SCRUM 모델을 채택한다.

#### 2.1.2 분석·설계 방법론

CV 모듈, NLP 모듈, UI 레이어, 데이터 관리 등 독립적 컴포넌트 단위의 개발·테스트·유지보수가 가능하도록 객체지향 방법론(OOP)을 채택한다. 모듈화된 구조는 향후 번역, TTS 등 기능 확장에도 용이하다.

### 2.2 WBS (Work Breakdown Structure)

| 대분류 | 중분류 | 세부 활동 |
|--------|--------|-----------|

## 3. 팀 구성

### 3.1 팀 구조

팀장의 역할은 존재하지만 모든 멤버가 상호 협력하고 서로의 개발 활동을 확인하며 진행하는 수평적 팀 구조를 채택한다. 각 담당 모듈에 대해 개별 책임을 가지되, 주요 의사결정은 전체 팀원 합의를 원칙으로 한다.

## 4. 품질 관리

### 4.1 팀 미팅 계획

| 미팅 종류 | 시간 | 토의 내용 | 참석자 |
|-----------|------|-----------|--------|
| Sprint Planning Meeting | 매주 금요일 18:00 | Sprint 목표 및 Sprint Backlog 계획 | 팀원 전체, 팀장 |
| Daily Scrum Meeting | 매일 18:30 (30분) | 어제 한 일, 오늘 할 일, 장애 현상 공유 | 팀원 전체, 팀장 |
| Sprint Review | Sprint 종료 시점 | Sprint 기간 동안 팀원 전체가 각자 한 일 리뷰 | 팀원 전체, 팀장 |
| Sprint Retrospective | Sprint 종료 시점 | 개선할 것, 잘한 것, 새로 시작할 것 도출 | 팀원 전체, 팀장 |
| 비정기 회의 | 필요 시 수시 | 프로젝트 진행 문제 발생 시 대응 논의 | 팀원, 팀장, 지도교수(필요 시) |

### 4.2 변경 사항 관리 방법

- 기능 변경을 위해 팀원 전체가 권한을 가져야 하며 다수결 합의가 필요하다.
- 변경 사항이 발생되는 경우 변경 전·후에 대해 반드시 기록물로 남긴다.
- 관련 산출물은 팀원이 전부 소지한다.
- 의견 충돌로 진행이 멈출 시에는 팀장이 최종 결정권을 갖는다.
- 변경 사항이 결정되면 지도교수에게 반드시 보고하고, 필요 시 재변경될 수 있다.

### 4.3 산출물 관리 방법

#### 4.3.1 산출물 명명 방법 (Naming)

- 소프트웨어 버전: `[버전]_[해당 버전의 산출 일자]`  
  (예: `v1_2026_04_24.apk`)
- 문서: `[팀명]_[문서명]_[YYMMDD]_Doc-[버전]`  
  (예: `SilentTalk_프로젝트관리계획서_260424_Doc-001.docx`)
- 그 외 파일: 해당 파일의 목적이 명확하게 드러나는 이름  
  (예: `260424_cv_model.pt`)

#### 4.3.2 산출물 버전 관리 방법

- 소프트웨어: 파일명 맨 앞에 버전을 기재한다. (담당자: 팀원 전체)
- 문서: 파일명 맨 뒤에 날짜를 기재하고 문서 내 제·개정 이력을 수정한다.
- 그 외 파일: 파일명 앞에 날짜를 기재하며 필요 시 일련번호를 추가한다.

#### 4.3.3 산출물 저장 방법

- 소프트웨어: GitHub (버전 관리)
- 문서: 팀원 공유 클라우드 서버 (하드웨어 백업 병행)
- 그 외 파일: 팀원 공유 클라우드 서버 (하드웨어 백업 병행)
